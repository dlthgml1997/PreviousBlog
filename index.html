[백준알고리즘] 7576: 토마토 (파이썬)

2021-01-10
  • 문제 링크: https://www.acmicpc.net/problem/7576

풀이

시간 초과의 늪에 빠졌던 문제다. 한 박스에 익은 토마토가 2개 이상일 수 있는 점을 유의해서 bfs를 구현하는 점이 어려웠다.

  • 먼저 첫 날, 익은 토마토(1)의 좌표를 모두 apples에 담는다.

  • bfs를 실행한다.

    • day 를 -1 로 초기화한 이유

      1. 먼저 예외 상황을 생각해보자. 만약 모든 박스가 1과 -1로 채워져 있는 경우에는 주변을 익게 하지 않고도 모든 사과가 익어있는 상황이기 때문에 0이 프린트 되어야한다. 그러나 apples에는 1의 좌표가 모두 담겨있기 때문에 bfs 함수의 while이 한번은 실행될 것이다. 그렇기 때문에 day는 -1로 초기화 해준다.

      2. 모든 사과가 익은 날에 마지막 익은 사과는 자신 주변에 안익은 사과가 없어도 apples에 좌표가 담긴다. 그렇기 때문에 마지막 while문이 실행되고 이때 day 가 + 1 되기 때문에 처음에 day를 -1로 초기화해야 정상적인 day 가 출력될 수 있다.

    • 하루마다 전날 익은 사과는 apples에서 비우고 새롭게 익은 사과들을 apples에 담기도록 구현하기 위해 apples의 길이만큼 for문을 실행한다.

    • 모든 1주변 사과를 익게 만든 후, 안익은 사과가 있는지 확인하고 안익은 사과가 있다면 -1을 출력한다.

중요 포인트 !

  • 기존에 사용해왔던 queue = [] , queue.pop(0) 방식으로 하면 O(n)으로 느려지기 때문에 collections의 deque로 구현해 O(1) 이 되도록 해야한다.

시간 초과 다른 풀이

  • 이 풀이는 테스트 케이스는 맞았지만 시간 초과가 난 풀이이다.

    • 하루마다 while 문이 실행되도록 해서 이중 포문을 실행하는 거부터 문제다 !
      • 하루마다 익은 토마토 주변 토마토를 익게하고 visited에 몇일 지났는지를 표시한다.
      • 그 다음 날, while 문이 실행될 때 visited가 전날인 토마토를 찾아 다시 주변 토마토를 익게하도록 했다.
    • while 문 종료 조건은 이중 포문을 다 돌았을 때, 한번도 주변 토마토를 익히지 않았을 때이다. (익힐 수 있는 토마토는 다 익힌 상태)
    • 첫날에 안익은 토마토 주변이 다 비어있으면 -1을 print하고 실행을 종료하는 check를 구현했다.

Java Programming - 객체지향 핵심 원리 (자바 Modifier와 생성자)

학습목표

  1. static, final, abstract 예약어의 기능 이해 및 활용
  2. 생성자의 기능 이해
  3. this 예약어의 의미와 활용 방법 학습

1. 자바의 Modifier

static

멤버 변수와 메서드 앞에 붙일 수 있는 modifier로서, 활용 방법을 제어함.

  • static 예약어가 붙지 않는 인스턴스 변수

    • 생성된 인스턴스마다 그 안에 클래스의 인스턴스 변수들이 포함된다.
    • 일반적인 멤버 변수를 인스턴스 변수라고 부른다.

  • static 예약어가 붙는 클래스 변수

    • 클래스로부터 생성된 인스턴스에 포함되지 않는 변수

    • 많은 인스턴스를 생성하더라도 메모리에 하나의 변수만 존재한다.

    • 객체를 생성하지 않고도 접근할 수 있는 변수

    • 가장 대표적인 클래스 변수로는 System 클래스의 out 변수가 있다.

  • 클래스 변수와 인스턴스 변수의 차이

    • 인스턴스 변수는 클래스로부터 객체가 생성될 때마다 객체의 변수들이 생성되지만 클래스 변수는 클래스로 부터 생성된 모든 객체들이 하나의 클래스 변수를 공유한다. 그렇기 때문에 클래스 변수는 객체가 생성될 때 메모리 영역을 할당하지 않고, 클래스가 로딩되는 과정에서 메모리에 한 번만 할당되는 멤버이다.
    • 각 객체가 동일한 값을 가지는 인스턴스 변수를 포함하면 메모리 관리에 악영향을 끼친다.
    • 클래스 변수는 값이 변경되는 경우, 클래스로부터 생성된 모든 객체에서 변경된 값을 사용할 수 있다.

  • 예시

    • 세 직원의 이름과 나이는 다르지만 회사이름은 S전자로 동일한 것을 볼 수 있다. 이때, 생성된 객체가 많을 수록 메모리를 불필요하게 사용하고 있다.

      이 문제를 클래스 변수로 해결할 수 있다.

    클래스 변수를 선언한 경우를 그림으로 표현하면 아래와 같다.

  • 클래스 메서드

    • 인스턴스를 생성하지 않고도 사용이 가능한 메서드

    • 대표적인 클래스 메서드로는 Integer클래스의 parseInt() 가 있다.

    • 클래스 메서드는 인스턴스 변수를 참조할 수 없다. (클래스 메서드는 인스턴스를 생성하지 않고 호출이 가능하지만, 인스턴스 변수의 경우 인스턴스를 생성해야만 메모리가 잡히기 때문이다.)

      즉, 클래스 메서드에서는 클래스 메서드 호출 할 수 있다.


final

  1. 변수

    final이 붙은 변수는 단 한 번 초기화가 가능하지만, 초기화 이후에는 값을 변경할 수 없는 상수이다.

    • 공통된 값을 가지는 변수를 클래스 변수로 선언하여 메모리를 효율적으로 사용할 때, 클래스 변수의 값을 변경할 수 있도록 허용하는 경우엔 static 예약어의 의미가 상실되므로, final 예약어를 결합하여 사용함으로써 초기화 된 값을 변경할 수 없게하는 것이 좋다.

    • final과 static 을 결합해 멤버 변수를 선언한 경우 이 상수의 이름은 대문자로 선언하고 여러 단어가 결합된 경우라면 단어와 단어 사이 ‘_’를 이용한다.

      예: COMPANY_NAME

  2. 메서드

    • final 예약어는 상속 구조에서 Overriding(메서드 재정의)를 금지하기 위해 사용될 수 있다.
      • 오버라이딩 : 상속 구조에서 부모 클래스의 메서드와 동일한 모양의 메서드를 자식 클래스에서 상속 받지 않고 새롭게 정의하는 것을 말한다.

  3. 클래스

    • 클래스 선언 시, 클래스 앞에 final 예약어를 추가하면 상속을 금지하는 의미이다.


abstract

클래스 선언부에서 abstract 사용 시 -> 추상 클래스

메서드를 선언할 때 abstract 사용 시 -> 추상 메서드

  • 추상 메서드

    메서드의 시그니처(리턴 타입, 메서드명, 매개변수)만 정의되고 구체적인 행위, 즉 블록({})부분은 정의되지 않는 메서드를 의미한다.

    • 메서드의 시그니처가 선언되어 있기 때문에 메서드의 기본적인 기능은 유추가 가능하다.

  • 추상 클래스

    추상 메서드를 포함하고 있다면 반드시 추상 클래스로 선언되어야 한다. 그러나 추상 클래스가 꼭 추상 메서드를 포함하고 있지 않을 수도 있다.

    • 추상 클래스와 상속의 결합

      • 상속 관계에서 부모 클래스가 추상 클래스라면 자식 클래스는 부모 클래스의 추상 메서드를 재정의(Overriding) 해야한다.


2. 생성자(Constructor)

생성자는 클래스로부터 객체를 생성할 때 호출되며, 객체의 멤버 변수를 초기화하는데 사용되는 메서드이다.

  • 생성자를 이용하여 생성된 객체의 멤버 변수에 적절한 값들을 설정할 수 있다. (코드 단순화)

  • 생성자의 특징

    • 반환형이 없다(void도 허용되지 않는다).

    • 생성자는 클래스와 같은 이름을 가지는 특별한 메서드이다.

    • 생성자는 이름은 같지만 매개변수를 다르게하여 여러 개를 중복 정의(Overloading)할 수 있다.

    • 생성자는 키워드 ‘new’와 항상 같이 사용된다.

      new 생성자()

    • 명시적으로 작성하지 않을 경우, 기본 생성자가 제공된다.

      • 기본 생성자: 매개변수가 없는 생성자로 일정한 규칙에 의해 자동 초기화된다.


3. this

생성자나 메서드의 매개변수 이름이 객체 변수의 이름과 같을 때 this 예약어를 사용해 이를 구별한다.

  • this 예약어 사용으로 멤버 변수와 매개변수 이름을 구분할 수 있고, 가독성을 높인다.

  • 클래스 몸체에는 클래스의 멤버가 선언된다.

접근 제한자 (Access Modifier)

접근 관련 modifier를 통해 객체지향 언어의 중요 특성 중 하나인 정보 은닉(Information Hiding)을 구현할 수 있다.

  • 접근 권한 예약어에는 public, protected, private, (default) 가 있다.

    • public : 모든 클래스에서 접근이 가능한 클래스임을 의미한다.
    • (default) : 접근 권한 예약어를 지정하지 않으면, 같은 패키지 내의 클래스에서만 접근이 가능하다.

  • 활용 방법 예약어에는 final,abstract, static, transient, volatile, syncronized, native 가 있다.

    • final : 자식 클래스를 가질 수 없는 클래스임을 의미한다.
    • abstract : 객체 생성이 불가능한 추상클래스를 의미한다.

  • 접근 제한자의 범위

    변수의 종류

    • 전역(Global) 변수 : 클래스 선언부 밑에 선언된 변수로 멤버 변수라고 칭한다. 여러 메서드에서 공통으로 사용할 수 있다.

3) 메서드 (Method)

메서드를 정의함으로써 코드가 간결해지고 재사용성이 높아진다.

3-1) 메서드 Modifier

메서드 Modifier를 이용해 적절하게 정보 숨김을 구현할 수 있다.

접근 권한을 나타내는 Modifier의 경우 그 쓰임이 멤버 변수와 동일하다.

  • 메서드 Modifier - 접근 권한
    • public : 모든 클래스에서 접근이 가능하다.
    • protected : 동일 패키지에 속하는 클래스와 하위 클래스 관계의 클래스에 의해 접근이 가능하다.
    • private : 클래스 내에서만 접근이 가능하다.
  • 메서드 Modifier - 활용 방법
    • final : Overriding 이 불가능한 메서드를 정의할 때 이용한다.
    • static : 클래스에 소속된 클래스 메서드를 의미하며, 클래스 생성 시 만들어진다.
    • abstract : 추상 메서드를 의미하며, 하위 클래스에 의해 구현된다.
    • syncronyzed: Thread의 동기화를 위한 메서드이다.

3-2) 메서드의 자동 구현

  • 정보 은닉의 기본

    • 멤버 변수들은 대부분의 경우 private 로 선언한다.
    • public 으로 지정한 메서드를 통해 멤버 변수에 접근하도록 클래스를 구현한다.

  • 이클립스에서는 private 변수에 대한 public Getter, Setter 메서드들을 자동으로 생성할 수 있는 기능을 제공한다.

    • Getter 메서드

      private 변수에 저장된 값을 리턴하는 getXXX() 메서드를 의미한다.

    • Setter 메서드

      private 변수에 값을 저장하는 getXXX() 메서드를 의미한다.

  • 이클립스에서 Getter / Setter 메서드 자동 구현

    1. private으로 접근 제한된 변수들을 가진 클래스 정의

    2. 작성된 소스에서 <Alt> + <Shift> + <S> 키를 누른 후, 셋톱 박스에서 Generate Getters and Setters를 누른다.

      위와 같은 메서드가 클래스 안에 생성된다.

  • toString() 메서드 자동 구현

    toString() 메서드는 생성된 객체의 상태를 쉽게 확인 가능하게 한다.

    1. 작성된 소스에서 <Alt> + <Shift> + <S> 키를 누른 후, 셋톱 박스에서 Generate toString()을 누른다.

자바 언어의 탄생

1991년

Green Project에서 가정용 전자기기에서 사용되는 작은 컴퓨터 언어를 디자인 했다.

이때,

  1. 작고 견고한 언어, 신뢰할 수 있는 코드
  2. 특정 디바이스에 종속되지 않아야 함

와 같은 특성을 고려해 Virtual Machine에서 동작하는 중간코드를 생성해 이식성이 높은 언어를 디자인 했다.

1992년

초기 자바 언어는 절차지향언어와 객체지향 혼용해 사용했던 언어인 C++ 를 기반으로 개발되었다. 그러다가 C++ 의 여러가지 한계에 부딪혀

완벽한 객체지향 언어인 Oak 가 개발되었고 제임스 고슬링에 의해 명명된 Oak는 Oak라는 프로그램이 존재했기 때문에 이후 JAVA로 변경 되었다.

1994년

World Wide Web의 발전과 Mosaic의 사용에 주목

자바를 기반으로 한 웹 브라우저 제작 (HotJava 브라우저로 발전함)

applet이라 불리는 자바의 기능을 보여주기 위해 브라우저가 중간코드(bytecode)를 해석할 수 있도록 만듦

1996년~ 현재

Netscape 2.0에서부터 자바가 지원됨

많은 업체들이 자바를 지원하게 됨

자바 버전 1.2가 출시되면사. 자바 2 플랫폼으로 명명됨.


자바 언어의 특징

  1. 단순

    C++의 전위처리, 포인터 사용, 다중 상속, 구조체, goto문 사용, 연산자 중복 기능 등을 제외

    가비지 컬렉터에 의한 자동 메모리 관리로 할당된 메모리 해제를 신경쓰지 않아도 된다.

  2. 객체지향(object-oriented)

    ​ 언어라는 객체를 사용해서 프로그램을 작성하도록 하는 언어

    -> 재사용성을 높일 수 있음.

    코드의 재사용을 넘어 객체의 재사용을 뜻하며, 프로그램 개발에 필요한 수많은 클래스 라이브러리들을 API로 제공함

  3. 분산 처리(Distributed)에 용이하다.

    분산 환경에서 TCP/IP 등의 프로토콜을 통해 효율적으로 실행할 수 있도록 설계된 언어이다.

    • TCP/IP 네트워크 기능 내장
    • HTTP/ FTP 등과 같은 프로토콜을 쉽게 사용할 수 있도록 라이브러리 제공
    • 서로 다른 컴퓨터 상의 객체들도 원격으로 호출하여 실행할 수 있는 원격 메서드 호출과 관련된 RMI(Remote Method Invocation)기능의 라이브러리 제공

  4. 인터프리터에 의해 실행됨

    .java 형태의 텍스트 파일 -> 컴파일러에 의해 자바가상머신에서 수행될 수 있는 중간코드로 컴파일된다. -> .class(클래스 파일 == 바이트 코드) -> 실제 실행을 위해 인터프리터에 의해 기계어로 변환 -> 컴퓨터에서 실행

    자바는 컴파일 언어임과 동시에 인터프리터 언어이다.

  5. 견고

    에러의 원인이 되는 요소들을 제거

    • 포인터를 사용하지 않음
    • 가비지 컬렉션
    • 데이터 타입 검사를 통해 에러 초기 검사
    • 실행시간에 발생하는 에러 초기

  6. 안전

    • 컴파일 시에 엄격하게 데이터 타입을 검사함으로써 프로그램 실행 시 발생할 수 있는 비정상적인 상황을 미리 감지

  7. 플랫폼 독립(Architecture Neutral)

    사용하는 운영체제나 CPU 등의 하드웨어 사양에 관계없이 실행될 수 있음.

    이식성이 높다.

  8. 높은 성능

    가비지 컬렉션 : 메모리가 할당된 객체의 상태를 추적 관리, 더이상 참조 되지 않으면 다종으로 할당된 메모리 해제 (불필요한 메모리 낭비 방지)

  9. 멀티스레드

    스레드 : 독자적으로 수행하는 하나의 작은 프로그램

    멀티스레드: 한번의 여러개의 스레드가 동시에 수행되는 과정 / 하나의 CPU가 여러 개의 프로그램을 동시에 수행하도록 함 (수행 속도를 빠르게 함)

  10. 동적

    변화하는 환경에 잘 적응되도록 설계됨

    기존의 프로그램에 영향을 주지 않고, 라이브러리에 새로운 메서드나 속성들을 추가할 수 있음

    프로그램과 프로그램에서 사용되는 라이브러리와의 연결을 실행시간에 수행함

    -> 라이브러리의 변화를 곧바로 적용할 수 있음

  • 문제 링크: https://programmers.co.kr/learn/courses/30/lessons/67256

풀이

정답 풀이

phone으로 정의한 딕셔너리를 이용해 좌표 값을 value로 두고, 거리를 구했다.

  • 좌표 상의 거리를 구하는 방법은 (x1 - x2)의 절댓값 + (y1- y2)의 절댓값이다.

    파이썬의 내장 함수 abs를 활용했다.

  • 딕셔너리는 key: value로 이루어져있고, key를 이용해 value를 얻는 방법은 value = 딕셔너리.get(key) 이다.

이렇게 쉽게 풀 수 있었던 것을 BFS로 푼다고 괜한 고생을 했다 ..!
문제 풀 때 딕셔너리 활용을 떠올리는 습관을 들여야겠다.


처음 푼 풀이 (런타임 에러)

BFS를 이용해 풀려고 했는데 런타임 에러가 났다.
두번째 열의 숫자가 나올때마다 이중 포문을 돌고 그 안에서 BFS를 두번이나 실행하기 때문이라고 판단했다.
